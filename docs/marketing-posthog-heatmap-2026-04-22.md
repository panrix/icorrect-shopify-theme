# Marketing Brief: PostHog Heatmap Artifacts
**Date:** 2026-04-22
**Author:** Marketing Jarvis
**Priority:** P1 — Data quality

---

## Problem

PostHog heatmaps show `script_placeholder` text instead of actual page content (headings, buttons, links). This makes heatmaps unusable for analyzing user behavior.

**Evidence:**
- PostHog heatmap view for any page shows `script_placeholder` in place of actual content
- Session recordings show the same placeholder text
- Live site inspection confirms the text does not appear to real users

**Root cause identified:**
PostHog captures the DOM snapshot before deferred JavaScript renders dynamic content. The theme uses `defer="defer"` on scripts, causing PostHog to capture placeholder states.

---

## Hypothesis

PostHog's session replay captures the initial HTML state before deferred scripts execute. The `script_placeholder` is PostHog's representation of script tags that haven't executed yet.

**Confirmed:** The text `script_placeholder` does not exist in any theme file. It is generated by PostHog's capture mechanism.

---

## Fix Options

### Option 1: Configure PostHog to exclude scripts from capture (Recommended)
Update PostHog initialization in `layout/theme.liquid` to mask script tags and placeholder elements.

**Tradeoffs:**
- Pro: Simple, one-line config change
- Pro: No impact on site performance
- Con: May mask some legitimate script-generated content

**Implementation:**
```javascript
posthog.init('phc_E4K3ulD49Fth2l3WOFjPpBbdgsbJwRMrErbWGjpppLB', {
  api_host: 'https://us.i.posthog.com',
  person_profiles: 'always',
  session_recording: {
    record_canvas: false,
    maskTextSelector: 'script, [data-ph-no-capture], .script-placeholder'
  }
});
```

### Option 2: Delay PostHog capture until page load
Configure PostHog to wait for `window.load` event before capturing snapshots.

**Tradeoffs:**
- Pro: Captures fully rendered page
- Con: May miss early user interactions
- Con: More complex implementation
